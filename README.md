## 一、可视化要回答的问题

> 本项目旨在探讨：**北京二手房的价格差异，究竟由什么驱动？地段与居住品质，这两种力量如何在不同空间圈层中此消彼长?**

具体拆解为以下三个子问题：

1. **空间分布**：北京各区的房价热力格局呈现何种特征？高价区与低价区在地理上如何分布？
2. **核心区悖论**：西城、东城单价最高，但这些高房价区域的居住品质（如房屋面积、电梯配备率）是否与价格相匹配？
3. **近郊特征对比**：五环外的近郊区域（如大兴、房山），其房源的面积与配套特征与核心区有何本质差异？

## 二、设计决策的原理
> 为了有效支持上述问题的探索，我们采用D3.js进行交互式可视化绘图。我们在视觉编码与交互技术上进行了针对性设计。

### 2.1 视觉编码策略

#### 颜色映射：D3 Plasma 色带（反向）
- **选择理由与设计逻辑**：我们采用的Plasma 是感知均匀的色带，亮度与色相呈连续变化，对色觉障碍用户友好。“高价=深色”的反向映射符合“深色即高值/重要”的直觉认知。
- **替代方案与取舍**：最初尝试使用 `YlOrRd`（黄→红），但在亮色地图背景上对比度较低；Viridis 的绿色辨识度也不佳。最终选择反向 `Plasma`，其深紫色与金黄色的色带在地图底图上对比度相对来说最好。

#### 地图底图：CartoDB Voyager
- **选择理由与设计逻辑**：作为亮色路网底图，Voyager 在缩放级别 12 以上能清晰显示道路和地名标签，为房源散点提供准确的地理参照背景。
- **替代方案与取舍**：CartoDB Dark为暗色地图，地图底色和二手房散点的对比度虽高，但路网信息较弱；标准 OpenStreetMap 信息过于密集，易干扰散点数据的阅读。

#### 散点大小与透明度
- **个体散点**：设置半径 4px，透明度 0.82，辅以白色细边框，确保在数据密集区保持可辨识性，防止颜色混叠。
- **聚合气泡**：半径按 `√count × 1.8` 缩放（14–44px），同时编码数量（大小）和均价（颜色），实现双变量视觉编码。

### 2.2 交互技术实现

#### 语义缩放（Semantic Zoom）
- **选择理由与设计逻辑**：由于原始数据量大，全局视角下逐点渲染缺乏意义。缩放级别小于 12 时，按行政区聚合为气泡，帮助用户快速感知宏观格局；放大后则散开为个体房源，支持微观维度的探索。
- **替代方案与取舍**：曾考虑 Leaflet.markercluster 插件，但其基于屏幕距离的聚合逻辑不符合本研究“按行政区划分析”的研究需求。

#### 双向范围滑块（noUiSlider）
- **选择理由与设计逻辑**：价价格和面积均为连续变量，用户需要同时控制上下界。单个滑块只能设定单一阈值，无法表达"我想看 50–100 ㎡的房子"这类区间查询。
- **替代方案与取舍**：两个独立的 `<input type="range">` 实现简单，但无法直观显示选中区间，且两端联动逻辑需要额外处理。

#### 多视图协同与焦点上下文 (Focus + Context)
- **选择理由与设计逻辑**：用户在地图上框选特定区域时，柱状图以灰色底色保留全北京数据（Context），同时用彩色叠加显示框选区域的局部数据（Focus），实现了局部与整体的无缝同屏对比。这是 Shneiderman 信息可视化 mantra（"overview first, zoom and filter, details on demand"）的直接实践。
- **替代方案与取舍**：考虑过在地图上直接显示区域统计标注，但会造成视觉拥挤；考虑过弹出式面板，但破坏了多视图联动的整体感。

#### 步进式叙事面板（Stepper Narrative）
- **选择理由与设计逻辑**：纯探索型可视化对初次用户门槛较高。步进式叙事提供"引导轨道"，通过 3 个逻辑递进的步骤帮助用户建立分析框架，最后一步解锁所有控件，过渡到自由探索模式。
- **替代方案与取舍**：考虑过 Scrollytelling（滚动叙事），但需要额外的滚动监听逻辑，且与地图交互存在冲突；考虑过固定注释（annotation），但无法动态改变地图视角和过滤状态。

#### 维度度量切换下拉菜单
- **选择理由与设计逻辑**：柱状图的 X 轴固定为行政区，但 Y 轴度量的选择直接决定分析视角。6 个维度（均价、总价、套数、面积、电梯率、地铁率）覆盖了价格、流动性、居住品质、交通配套四个核心分析维度，避免了多图并列造成的视觉负担。

### 2.3 布局决策

采用三栏布局：**叙事面板（200px）| 地图（flex 1.5）| 图表区（flex 1）**

- 叙事面板在步骤 5 后自动隐藏，将空间归还给地图。
- 图表区分为上下两个等高区域：柱状图（区域对比）+ 趋势图（时间维度），分别回答"空间差异"和"时间演变"两个问题。
- 顶部控制栏集中所有过滤控件，避免控件散落在各视图中造成认知负担。

## 三、通过交互式可视化对问题的解释
&emsp;&emsp;首先，针对主问题，我们小组认为北京二手房的价格差异，主要由地段和居住品质这两种因素来决定。通过缩小地图（也就是放大地图的呈现氛围），我们可以明显地看出以区为聚类的北京十三个区的房价高低与二手房数量的多少。在六大主城区（海淀、朝阳、丰台、石景山、东城、西城），地段对于房价的决定作用强于居住品质；在其他近郊或远郊区，反之同理。
&emsp;&emsp;针对子问题一，在空间分布格局上，由散点图可知，北京二手房市场呈现出的“单中心圈层递减”与“北重南轻”双重特征。以地理中心（东城区）为起点，房价由二环至五环呈现出明显的阶梯式下降。与此同时，高价房源并非围绕核心均匀分布，而是向北及西北方向倾斜。海淀与朝阳依托顶尖的教育资源与科创、金融产业集聚，形成了大面积的高价连片区；相对而言，大兴、房山等南城近郊以及顺义、门头沟等远郊区则构成了主要的低价空间洼地。
&emsp;&emsp;针对子问题二，我们发现，在房价最高的地理核心，存在着一个“核心区悖论”——价格与物理居住品质发生严重错配。在可视化图表中可以直观看到，尽管西城、东城的单价色值最深，但如果勾选“有电梯”以及查询其“套均面积”，会发现其居住品质维度的指标（如平均房屋面积、电梯配备率）却处于底层。这些区域的交易主力多为上世纪八九十年代的老旧板楼，户型狭小且极度缺乏电梯。这种现象揭示了核心区极高单价的本质——购房者支付的并非物理建筑的溢价，而是附着于地段上的稀缺城市公共资源（如顶配的医疗及极短的通勤时间与极高的便利度）。
&emsp;&emsp;针对子问题三，与核心区的“老破小”截然相反，五环外的近郊区域在数据上展现出了明显的“居住升级”特征。从动态条形统计图可以看出，大兴、房山等区域单价较低，而且房源多为近十几年内新建的商品房或高层公寓楼，不仅电梯配备率极高，主力成交户型也多为 80-110 平方米的中大居室。这种核心区与近郊区的本质差异，反映了当前市场刚需群体的一种空间置换策略，即用更长的通勤时间、相对更劣的地段去换取物理居住品质的提升与更现代化的社区生活环境。

## 五、开发过程概述
> 本项目由我们 3 人共同协作完成。为了在两周内搞定地图和图表的联动交互，我们将整个项目拆解为三个核心模块，大家各司其职，总共花费了大约 60 个小时（每人平均约 20 小时），也参考了一些既有文献和AI的指导： 

- 陈飞涵（投入约 20 小时）：负责Kaggle 数据集搜集与匹配交互式可视化的数据预处理工作,此外还负责提供一些灵感和细节/思路部分
    - 主要挑战与耗时点：Kaggle数据集由我搜集而来且是我的提议，选这个北京二手房价的数据集是因为房子的Feature比较丰富且适配于地图可视化，预期效果会比较好。原始的 Kaggle 北京二手房数据集有将近 32 万条记录，前端浏览器根本无法一次性渲染。这部分工作最耗时的地方在于空间坐标的清洗与分层抽样。我用 Python 写了脚本，不仅要剔除价格异常值，还要解决原始坐标系偏移的问题。为了让前端聚合地图（气泡图）能准确对应北京的行政区划，我根据经纬度重新做了区划映射，最后按各区真实房源比例，进行统计学上的分层抽样（保留至约 5,000 条），并导出为结构化的 JSON 供前端直接调用。数据的准确性是可视化的地基，前期在这里反复校验花了不少功夫。此外，我还提出了“焦点上下文（Focus + Context）”和“步进式叙事（Stepper Narrative）”的想法并给出了文字初稿，供我的其他队友来实现。我还负责了报告的部分写作部分。
- 宋文韬（投入约 20 小时）：负责 D3.js 核心图表开发与多视图联动
    - 主要挑战与耗时点：写图表联动逻辑是前端最头疼的部分。我主要负责右侧的柱状图和趋势图，以及实现地图框选和图表之间的“焦点上下文（Focus + Context）”实时联动。开发中经常遇到两边数据不同步的 Bug（比如在地图上重绘选框时，图表数据没有及时清空或更新）。后来我们重新梳理了数据过滤逻辑，写了一个统一的回调函数来管理组件间的状态传递，这才解决了交互冲突，保证了筛选时的流畅度。我也负责了报告的部分写作部分。
- 颜韬（投入约 20 小时）：负责 Leaflet 地图底图集成、步进式叙事与 UI 统筹
    - 主要挑战与耗时点：我起初在地图底图的选择上踩了坑，排查跨域报错后最终敲定了 CartoDB Voyager 亮色底图。最费时间的挑战在于开发“步进式叙事（Stepper Narrative）”。为了防止用户在引导流程中乱点导致页面错乱，我写了大量逻辑去精确控制不同步骤下地图平移、滑块和下拉菜单的“禁用/解锁”状态。同时他还负责了解决图层叠加引起的 CSS 遮挡问题，以及整份分析报告的书面化统筹。
